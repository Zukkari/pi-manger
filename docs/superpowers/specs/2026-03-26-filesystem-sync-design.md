# Pi Manager — Filesystem Sync Design

**Date:** 2026-03-26

## Overview

Sync the mounted filesystem into a SQLite database on startup and every minute thereafter. Each file and directory gets a stable internal ID, enabling a future tree view API. The sync is a background concern — the HTTP server stays up regardless of sync errors.

---

## Goals

- Walk `MANAGED_DIR` recursively and persist metadata (id, parent_id, name, path, size, is_dir, modified_at) to SQLite.
- Run once at startup, then on a 1-minute ticker.
- Hard-delete DB rows for paths no longer present on disk.
- Keep file IDs stable across syncs (same path → same ID).

| `PORT`        | No       | `8080`             | HTTP server port (unchanged)             |
| `DB_PATH`     | No       | `./pi-manager.db`  | Path to the SQLite database file         |

In Docker Compose, `DB_PATH` points to a named volume so the database persists across container restarts (see docker-compose changes below).

## Database Schema

```sql
CREATE TABLE IF NOT EXISTS files (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    parent_id   INTEGER REFERENCES files(id) ON DELETE CASCADE,
    path        TEXT NOT NULL UNIQUE,
    name        TEXT NOT NULL,
    size        INTEGER NOT NULL DEFAULT 0,
    is_dir      INTEGER NOT NULL DEFAULT 0,
    modified_at INTEGER NOT NULL,
    synced_at   INTEGER NOT NULL
);
```

- `path` is the upsert key — the same path always maps to the same `id`.
- `parent_id` is `NULL` for the root. `ON DELETE CASCADE` removes children when a directory row is deleted.
- `modified_at` stores the file's `mtime` as a Unix timestamp (int64 seconds). Linux has no reliable birth time; INTEGER avoids SQLite driver timezone/parsing complexity.
- `synced_at` records when the row was last written as a Unix timestamp — useful for debugging.

## SQLite Driver

`modernc.org/sqlite` — pure Go, no CGO. Cross-compiles cleanly to `linux/arm64` in the existing Dockerfile without any changes to the build stage.

---

## sqlc

Type-safe Go query code is generated from `store/query.sql` via `sqlc generate`. Generated file `store.gen.go` is committed so the build requires no external tools at runtime.

`sqlc.yaml`:
```yaml
version: "2"
sql:
  - engine: "sqlite"
    queries: "query.sql"
    schema: "schema.sql"
    gen:
      go:
        package: "store"
        out: "."
```

Queries in `query.sql`:
```sql
-- name: UpsertFile :one
INSERT INTO files (parent_id, path, name, size, is_dir, modified_at, synced_at)
VALUES (?, ?, ?, ?, ?, ?, ?)
ON CONFLICT(path) DO UPDATE SET
    parent_id   = excluded.parent_id,
    name        = excluded.name,
    size        = excluded.size,
    is_dir      = excluded.is_dir,
    modified_at = excluded.modified_at,
    synced_at   = excluded.synced_at
RETURNING id;

-- name: DeleteMissing :exec
DELETE FROM files WHERE path NOT IN (/*SLICE:paths*/);
```

## Internal Design

### `internal/store`

- `Open(dbPath string) (*Store, error)` — opens the SQLite file, runs `CREATE TABLE IF NOT EXISTS` migration, returns a `*Store`.
- `*Store` wraps the sqlc-generated `*Queries` and exposes `UpsertFile` and `DeleteMissing`.

### `internal/scanner`

Exposes:
```go
func Sync(ctx context.Context, root string, s Store) error
```

Where `Store` is a local interface (scanner imports `store` for the params type):
```go
type Store interface {
    UpsertFile(ctx context.Context, arg store.UpsertFileParams) (int64, error)
    DeleteMissing(ctx context.Context, paths []string) error
}
```

Logic:
1. `filepath.WalkDir(root, ...)` — visits all entries depth-first.
2. Track a `map[string]int64` of `path → id` built during the walk. Parent is always visited before children, so `parent_id` is always resolvable at insertion time.
3. `UpsertFile` for each entry; store returned `id` in the map.
4. After the walk, `DeleteMissing(seenPaths)`.
5. Per-file walk errors (e.g., permission denied) are logged and skipped; the walk continues.

### `cmd/api/main.go` changes

```go
// Open DB
db, err := store.Open(dbPath)
if err != nil {
    log.Fatalf("failed to open database: %v", err)
}

// Startup sync
runSync(ctx, managedDir, db)

// Background ticker
go func() {
    ticker := time.NewTicker(time.Minute)
    defer ticker.Stop()
    for {
        select {
        case <-ticker.C:
            runSync(ctx, managedDir, db)
        case <-ctx.Done():
            return
        }
    }
}()
```

`runSync` logs start, calls `scanner.Sync`, logs completion or error.

## Error Handling

| Scenario | Behavior |
|---|---|
| `DB_PATH` not writable / DB fails to open | Server exits at startup with clear message |
| Startup sync fails | Logged as warning; server starts, disk endpoint still works |
| Ticker sync fails | Logged; next tick retries |
| Single file unreadable during walk | Logged and skipped; walk continues |
| `MANAGED_DIR` disappears between syncs | Walk error logged; DB retains last known state |

## Docker Compose Changes

`docker-compose.yml` gains a `DB_PATH` env var and a named volume for the SQLite file:

```yaml
services:
  backend:
    build:
      context: ./backend
    platform: linux/arm64
    expose:
      - "8080"
    environment:
      - MANAGED_DIR=/data
      - PORT=8080
      - DB_PATH=/db/pi-manager.db   # new
    volumes:
      - ${PI_MANAGED_DIR:-/tmp}:/data
      - db-data:/db                  # new
    restart: unless-stopped

  frontend:
    build:
      context: ./frontend
    platform: linux/arm64
    ports:
      - "80:80"
    depends_on:
      - backend
    restart: unless-stopped

volumes:             # new
  db-data:
```

The `db-data` named volume is managed by Docker and persists across `docker compose down` / `up` cycles. The SQLite file lives at `/db/pi-manager.db` inside the container.
