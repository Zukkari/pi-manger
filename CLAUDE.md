## Project Overview

**pi-manager** is a full-stack app for monitoring Raspberry Pi resources. The backend is a Go API server that syncs a managed directory into SQLite and exposes HTTP endpoints. The frontend is a React + TypeScript + Vite app that visualizes disk and file data.

## Commands

### Backend (Go)

```bash
cd backend
go build ./...                          # Build all packages
go build -o api ./cmd/api               # Build the server binary
go run ./cmd/api                        # Run server (requires env vars)
go test ./...                           # Run all tests
go test ./internal/store/...            # Run a specific package's tests
```

Required env vars when running locally:
```
MANAGED_DIR=/path/to/monitor    # Required
PORT=8080                        # Optional, default 8080
DB_PATH=./pi-manager.db          # Optional
```

#### sqlc (code generation)

SQL queries are in `backend/internal/store/query.sql`. After editing them, regenerate Go code:
```bash
cd backend && sqlc generate
```
Do not manually edit `query.sql.go` ‚Äî it is generated by sqlc.

### Frontend (React)

```bash
cd frontend
npm install
npm run dev        # Dev server with HMR
npm run build      # TypeScript check + Vite bundle
npm run lint       # ESLint
npm run test       # Vitest
```

### Docker (full stack)

```bash
docker-compose up --build    # Build and run backend + frontend
```

Targets `linux/arm64` (Raspberry Pi). Backend serves on port 8080, frontend on port 80 via Nginx.

```
    ‚îú‚îÄ‚îÄ scanner/             # Filesystem walk + DB sync (runs at startup + every minute)
    ‚îî‚îÄ‚îÄ store/               # SQLite layer via sqlc-generated code
        ‚îú‚îÄ‚îÄ schema.sql       # Single `files` table definition
        ‚îú‚îÄ‚îÄ query.sql        # sqlc source queries
        ‚îî‚îÄ‚îÄ query.sql.go     # Generated ‚Äî do not edit
```

**Data flow**: `scanner.Sync` walks `MANAGED_DIR` recursively ‚Üí calls `store.UpsertFile` per entry ‚Üí calls `store.DeleteMissing` to prune stale records. Runs at startup and every 60s.

**Database schema** ‚Äî `files` table:
- Hierarchical: `parent_id INTEGER REFERENCES files(id) ON DELETE CASCADE`
- `path TEXT NOT NULL UNIQUE` ‚Äî absolute path
- Timestamps: `modified_at` (from filesystem), `synced_at` (last scan), both Unix seconds

**Frontend conventions**:
- Always use named exports ‚Äî never `export default`

**Frontend patterns**:

This is a dashboard-style app. The key rules:

- **Components** are the base building blocks. Each component fetches its own data (via React Query) and handles its own loading/error states independently ‚Äî a failing widget must not block the rest of the page. Components are designed to be reusable across pages.
- **Layouts** arrange components using HTML structure. They may contain base HTML elements but no business logic unless explicitly specified.
- **Pages** are composed exclusively of layouts and components ‚Äî no raw HTML elements. Pages are oblivious to what data their components load; they only decide which components and layouts to render.
- API calls live in `features/<name>/api/`, React Query hooks in `features/<name>/queries/`.

### API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/disk` | Disk usage stats (`total_bytes`, `used_bytes`, `free_bytes`, `used_percent`) |
| GET | `/api/files` | List files; optional `?parent_id=<id>`, omit for root entries |
